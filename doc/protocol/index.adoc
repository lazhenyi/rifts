This section documents the Rift/1 wire protocol as implemented by the `rifts` crate.

== Binary Frame Envelope (spec §6)

Every frame on the wire is a 24‑byte header followed by an N‑byte payload:

[source,text]
----
 0      1      2      3      4             11     12            19     20     23
+------+------+------+------+-------------+------+-------------+------+------+
| type | codec| flags (u16 BE) | frame_id   (u64 BE) | timestamp   (u64 BE) |
+------+------+------+------+-------------+------+-------------+------+------+
 20     23      24              23 + N
+------+------+------+// .... //+------+
| plen (u32 BE) |  payload  ...   |
+------+------+------+// .... //+------+
----

=== Field Reference

[cols="3m,3m,7a",options="header"]
|===
| Field | Type | Description

| `type`
| u8
| Frame type tag: `C` (Control), `D` (Data), `A` (Ack), `F` (Flow), `E` (Error).

| `codec`
| u8
| Payload encoding: `J` (JSON), `B` (CBOR).

| `flags`
| u16 BE
| Bitmask (see <<Flags>>).

| `frame_id`
| u64 BE
| Monotonic frame ID within the connection.

| `timestamp`
| i64 BE
| Sender timestamp (ms since epoch).

| `plen`
| u32 BE
| Payload length in bytes.

| `payload`
| bytes
| Codec‑encoded business or control data.
|===

[[Flags]]
=== Frame Flags (spec §6.3)

[cols="2m,1m,5a",options="header"]
|===
| Name | Bit | Description

| `COMPRESSED`
| 0
| Payload is compressed.

| `ENCRYPTED`
| 1
| Payload is application‑layer encrypted.

| `FRAGMENTED`
| 2
| Frame is a fragment (not the last).

| `FINAL_FRAGMENT`
| 3
| Last fragment in a sequence.

| `REQUIRES_ACK`
| 4
| Peer must send an ack.

| `REPLAYED`
| 5
| Frame comes from a replay buffer.

| `SNAPSHOT`
| 6
| Payload is a snapshot.

| `DEGRADED`
| 7
| Message was downgraded by the server.

| `DUPLICATE`
| 8
| Server detected a duplicate.

| `TRACE`
| 9
| Frame carries distributed‑trace metadata.
|===

== Message Classes (spec §8.1)

[cols="3m,3m,4a,",options="header"]
|===
| Class | Default Delivery | Use Case

| `event`
| `at_least_once`
| Chat, notifications, business events.

| `command`
| `at_least_once`
| RPC, operation requests.

| `reply`
| follows command
| Command responses.

| `state`
| `latest_only`
| Online presence, cursor position, room counts.

| `datagram`
| `best_effort`
| Game input, typing indicators, telemetry.

| `stream`
| `durable_ordered`
| AI token streaming, file transfer, media chunks.

| `snapshot`
| `at_least_once`
| Reconnect re‑initialisation.

| `system`
| `at_least_once`
| Rate‑limit notices, migration, maintenance.

|===

[[Delivery]]
== Delivery Modes (spec §8.2)

[cols="3m,7a",options="header"]
|===
| Mode | Guarantee

| `at_most_once`
| Fire‑and‑forget. May be lost. No retry.

| `at_least_once`
| May duplicate. Server assigns offset.  Client dedupes with `message_id`.

| `exactly_once_effect`
| Protocol provides idempotency key + dedupe window.  Business layer guarantees side‑effect‑once.

| `latest_only`
| Older values are overwritten by newer ones on the same `state_key`.

| `best_effort`
| No ack, no replay.  Dropped under backpressure.

| `durable_ordered`
| Persisted, strictly ordered, replayable by offset.
|===

[[Ack]]
== Acknowledgement (spec §12)

Ack statuses exchanged between client and server:

[cols="2,6a",options="header"]
|===
| Status | Meaning

| `received`
| Peer received the frame.

| `accepted`
| Server accepted the message for processing.

| `persisted`
| Message is durably stored.

| `delivered`
| Message reached the target subscriber(s).

| `processed`
| Business handler completed successfully.

| `rejected`
| Message was rejected (permission, validation).

| `expired`
| TTL elapsed before delivery.

| `duplicate`
| Dedupe key collision — not re‑executed.

| `failed`
| Processing failed.
|===

[[Err]]
== Error Codes (spec §19.1)

=== Protocol

[horizontal]
`RIFT_PROTOCOL_VERSION_UNSUPPORTED`::  Client/server version mismatch.
`RIFT_PROTOCOL_FRAME_INVALID`::       Malformed frame.
`RIFT_PROTOCOL_CODEC_UNSUPPORTED`::   Codec not in server’s supported set.
`RIFT_PROTOCOL_PAYLOAD_TOO_LARGE`::   Payload exceeds `max_payload_bytes`.
`RIFT_PROTOCOL_REQUIRED_FIELD_MISSING`::  Required field absent.
`RIFT_PROTOCOL_SCHEMA_MISMATCH`::     Schema version not compatible.
`RIFT_PROTOCOL_ORDER_VIOLATION`::     Message order constraint broken.

=== Auth & Permission

[horizontal]
`RIFT_AUTH_REQUIRED`::      Anonymous not permitted.
`RIFT_AUTH_INVALID`::       Token / credential rejected.
`RIFT_AUTH_EXPIRED`::       Session or token past expiry.
`RIFT_AUTH_REVOKED`::       Token actively revoked.
`RIFT_PERMISSION_DENIED`::  Operation not authorised.
`RIFT_TOPIC_FORBIDDEN`::    Topic access denied by policy.

=== Session & Resume

[horizontal]
`RIFT_SESSION_NOT_FOUND`::      Session ID unknown.
`RIFT_SESSION_EXPIRED`::        Resume window closed.
`RIFT_SESSION_CONFLICT`::       Epoch mismatch.
`RIFT_RESUME_REJECTED`::        Resume attempt denied.
`RIFT_REPLAY_OFFSET_EXPIRED`::  Offset outside replay window.
`RIFT_SNAPSHOT_REQUIRED`::      Client must pull a fresh snapshot.

=== Topic

[horizontal]
`RIFT_TOPIC_NOT_FOUND`::         Topic does not exist.
`RIFT_TOPIC_CLOSED`::            Topic has been closed.
`RIFT_TOPIC_OVERLOADED`::        Topic is under backpressure.
`RIFT_TOPIC_SUBSCRIBER_LIMIT`::  Maximum subscribers reached.
`RIFT_TOPIC_PUBLISHER_LIMIT`::   Maximum publishers reached.
`RIFT_TOPIC_RATE_LIMITED`::      Per‑publisher or per‑topic rate exceeded.

=== Message

[horizontal]
`RIFT_MESSAGE_DUPLICATE`::       Duplicate `message_id` or `dedupe_key`.
`RIFT_MESSAGE_EXPIRED`::         TTL elapsed.
`RIFT_MESSAGE_REJECTED`::        Validation or policy rejection.
`RIFT_MESSAGE_TOO_LARGE`::       Payload exceeds limit.
`RIFT_MESSAGE_ACK_TIMEOUT`::     Expected ack not received.
`RIFT_MESSAGE_DELIVERY_FAILED`:: Fanout or subscriber delivery failure.

=== System

[horizontal]
`RIFT_SYSTEM_OVERLOADED`::         Node is overloaded.
`RIFT_SYSTEM_MAINTENANCE`::        Scheduled maintenance.
`RIFT_SYSTEM_SHARD_MOVED`::        Topic shard relocated.
`RIFT_SYSTEM_REGION_UNAVAILABLE`:: Region temporarily unavailable.
`RIFT_SYSTEM_INTERNAL`::           Unexpected internal error.

== Topic Profiles (spec §9.2)

Each topic carries a `TopicProfile` with these policies:

[cols="2,6a",options="header"]
|===
| Policy | Options

| `retention`
| `None` / `Ttl(Duration)` / `Count(usize)` / `Size(usize)` / `Durable` / `Latest`

| `ordering`
| `None` / `Connection` / `Publisher` / `Topic` / `Key` / `Causal`

| `auth_policy`
| Per‑action permission callback.

| `fanout_policy`
| `Direct` (small topics, default) / `TreeFanout` / `RegionalFanout` / `EdgeCache`.

| `rate_limit`
| Per‑publisher + per‑topic token‑bucket (messages/sec).

| `backpressure`
| Strategy selection: pause, drop‑volatile, coalesce‑state, downgrade, disconnect.

| `replay`
| Whether offsets are stored and replayable.  Window: `replay_window`.

| `snapshot`
| Whether periodic snapshots are captured.
|===

== Topic Naming (spec §9.1)

- UTF‑8, max 256 bytes.
- No control characters.
- Must not start with `$` (reserved for system topics).
- Recommended convention: `/`‑delimited hierarchy (`room/{id}`, `user/{id}`, `org/{id}`).

== Heartbeat Policy (spec §21)

The server sends heartbeat parameters during the `ready` handshake.  The client
must send a ping within `ping_interval_ms`; the server replies with a pong.  If
`max_missed_pongs` consecutive pongs are missed, the connection is closed with
`CloseCode::HeartbeatTimeout`.

[cols="3,5a",options="header"]
|===
| Parameter | Default (spec §27.1)

| `ping_interval_ms`
| 25 000

| `pong_timeout_ms`
| 10 000

| `max_missed_pongs`
| 2

| `idle_timeout_ms`
| 300 000

| `jitter_ms`
| 2 500
|===

The `jitter_ms` value is a hint to the client — it should randomise its ping
interval by ±`jitter_ms` to avoid thundering‑herd heartbeat storms.

== Version Negotiation (spec §4)

The protocol version is a 16‑bit unsigned integer encoded as `major * 256 + minor`.
Rift/1.0 = 256.  During the `hello` handshake the client sends its version; the
server replies with the negotiated version in `welcome`.  If the server cannot
support the client's version, it closes with
`RIFT_PROTOCOL_VERSION_UNSUPPORTED`.

== Subscribe Modes (spec §9.3)

[cols="2,6a",options="header"]
|===
| Mode | Behaviour

| `live`
| Receive only messages published after the subscription is accepted.

| `replay`
| Receive messages from a given offset forward.  Requires `replay_enabled` on the
  topic profile.

| `snapshot_then_live`
| Server sends the latest snapshot (if available), then switches to live delivery.
  Used for re‑initialisation after reconnect.

| `last_value`
| Server sends only the most recent retained message, then no further messages.
|===

== Priority Levels (spec §8.3)

[cols="2,1,6a",options="header"]
|===
| Level | Tag | Treatment

| `Critical`
| 0
| Always delivered.  Never dropped under backpressure.  Ack required.

| `High`
| 1
| Delivered ahead of normal.  Dropped only under extreme pressure.

| `Normal`
| 2
| Default priority.  Standard FIFO delivery.

| `Low`
| 3
| Delivered after normal.  Eligible for coalescing.

| `Volatile`
| 4
| May be dropped silently under backpressure.  No ack, no replay.

| `Background`
| 5
| Bulk data, analytics.  Lowest priority.  Dropped first.
|===

=== Implementing a Broker Node

Any language that can speak framed CBOR over TCP can implement the broker side.
The expected behaviour per request:

- `Publish` → validate, dedupe, allocate offset, append log, fanout to subscribers,
  return `PublishResult`.
- `Subscribe` → register the sink, return `SubscribeResult` with an id.
- `Deliver` (push) → sent from broker to gateway when a subscribed topic receives
  a message.  The gateway routes it to the local `FanoutSink`.
